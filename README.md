# Synchronous FIFO: UVM Verification

This project implements a UVM test environment for a parametrized synchronous FIFO buffer. As
opposed to a standard testbench, a UVM testbench improves efficiency through a standardized
methodology and allows reusability for multiple tests in the future.

## DUT Overview

* Parametrized FIFO data width (`WIDTH`) and FIFO depth (`DEPTH`)
* Single clock for both write and read operations
* Support for simultaneous write and read operations
* Synchronous active-low reset

## Testbench Overview

The implemented test is a stress-test-style constraint-random verification that runs 2,000
transactions using UVM 1.2. The test was run on EDA Playground using the Cadence Xcelium simulator.

Click on this link to view the test: https://www.edaplayground.com/x/YmYC.

### Inclusions

| Component            | Role                                                                                      |
| -------------------- | ----------------------------------------------------------------------------------------- |
| `fifo_if`            | Interface bundling DUT inputs/outputs and internal pointers between testbench and DUT     |
| `fifo_sequence_item` | Transaction class defining randomized DUT inputs and captured DUT outputs                 |
| `fifo_sequence`      | Generates a stream of randomized `fifo_sequence_item` transactions for the test           |
| `fifo_sequencer`     | Arbitrates and forwards sequence items from the sequence to the driver                    |
| `fifo_driver`        | Drives transaction values from the sequencer onto the DUT via the virtual interface       |
| `fifo_monitor`       | Samples DUT inputs/outputs each clock cycle and broadcasts them via its analysis port     |
| `fifo_agent`         | Encapsulates and connects the driver, monitor, and sequencer                              |
| `fifo_scoreboard`    | Reference model that compares expected vs. actual DUT outputs to check correctness        |
| `fifo_coverage`      | Covergroup-based subscriber that tracks functional coverage of corner-case scenarios      |
| `fifo_env`           | Instantiates and connects the agent, scoreboard, and coverage components                  |
| `fifo_test`          | Builds the environment and starts the sequence to run the test                            |
| `fifo_tb_top`        | Top-level module instantiating the interface, DUT, clock generation, and running the test |

### Constraints

To emulate desired behavior, two constraints were applied during the randomized stimulus generation:

* Resets will occur very infrequently to allow the DUT to perform write and/or read operations
  uninterrupted.

* The DUT will frequently write or simultaneously write and read. This is to allow the
  DUT to fill up the FIFO space more easily and to ensure that the simultaneous operation occurs
  frequently enough to further verify its functionality.

### Scoreboard

The scoreboard includes a queue-based reference model, which will use the provided inputs to
calculate the expected outputs and compare them to the DUT outputs.

To implement this method of verification, the scoreboard needs the inputs from a given clock cycle
and the outputs generated from those inputs, which will be reflected in the next clock cycle.
However, the monitor observes inputs and outputs from a single given clock cycle, which would mean
that those inputs are actually new inputs (and the outputs from those new inputs have yet to be
generated).

To handle this mismatch, the scoreboard creates a "delay" when using the inputs by storing given
inputs to use them for the next monitor-provided outputs. This will allow the reference model to use
the "previous" inputs and the "current" outputs (which are generated by the "previous" inputs).

#### Reference Model Operations

* To perform a reset operation, the model simply empties the queue.
* To perform a read operation, the model pops the first element of the queue and stores the value
  in a variable.
* To perform a write operation, the model pushes the new element to the back of the queue.
* To update the full and empty flags, the model uses the size of the queue.

### Functional Coverage

The table below lists the corner-case scenarios that were accounted for by the coverage model. The
printed end-of-test coverage report will also list each scenario and indicate either `100%` if the
scenario was covered in the test or `0%` otherwise.

| Corner-Case Scenario                              | Expected Behavior                 |
| ------------------------------------------------- | --------------------------------- |
| Reset when FIFO is full                           | `full = 1 -> 0`, `empty = 0 -> 1` |
| Reset when FIFO is empty                          | `full = 0 -> 0`, `empty = 1 -> 1` |
| Write when FIFO is full                           | `full = 1 -> 1`, `empty = 0 -> 0` |
| Write when FIFO is empty                          | `full = 0 -> 0`, `empty = 1 -> 0` |
| Read when FIFO is full                            | `full = 1 -> 0`, `empty = 0 -> 0` |
| Read when FIFO is empty                           | `full = 0 -> 0`, `empty = 1 -> 1` |
| Write last element to make FIFO full              | `full = 0 -> 1`, `empty = 0 -> 0` |
| Read last element to make FIFO empty              | `full = 0 -> 0`, `empty = 0 -> 1` |
| Write and read simultaneously when FIFO is full   | `full = 1 -> 0`, `empty = 0 -> 0` |
| Write and read simultaneously when FIFO is empty  | `full = ? -> 0`, `empty = ? -> 1` |
| Reset, write, and read simultaneously             | `full = ? -> 0`, `empty = ? -> 1` |
| Read pointer wraparound                           | `rd_ptr = DEPTH - 1 -> 0`         |
| Write pointer wraparound                          | `wr_ptr = DEPTH - 1 -> 0`         |

In addition to the above corner cases, the coverage model accounts for an illegal state (`full = 1`,
`empty = 1`). If the DUT reaches this state at any point in the test, the simulation will print an
error message.
